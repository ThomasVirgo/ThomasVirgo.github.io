HTTP & HTTP Methods

HTTP (Hypertext Transfer Protocol) is the request/response protocol that the web is built on. It is an application-layer (layer 7) protocol that, for HTTP/1.1 and HTTP/2, runs on top of TCP (see the TCP & UDP note). TCP gives HTTP a reliable, ordered byte stream between a client and a server; HTTP defines the grammar of what gets sent over that stream: a client asks for or acts on a resource, and the server answers.

The defining property of HTTP is that it is stateless. The server keeps no memory of previous requests. Every request must carry everything needed to understand it, which is why authentication tokens, cookies, and content negotiation all ride along in headers on each request. Statelessness is what lets any server behind a load balancer handle any request, which is the foundation of horizontal scaling.

Anatomy of a request and response

An HTTP request is plain text (in HTTP/1.1) made of a request line, headers, a blank line, and an optional body:

POST /api/users HTTP/1.1
Host: example.com
Content-Type: application/json
Content-Length: 27
Authorization: Bearer abc123

{"name": "Tom Virgo"}

The response mirrors that shape: a status line, headers, a blank line, and an optional body:

HTTP/1.1 201 Created
Content-Type: application/json
Location: /api/users/42

{"id": 42, "name": "Tom Virgo"}

So every exchange has three moving parts worth keeping straight: the method (what action), the status code (how it went), and the headers (the metadata that drives caching, auth, content type, and connection handling).

Methods

The method (also called the verb) states the intent of a request. The common ones:

  • GET. Retrieve a resource. No body, no side effects.
  • POST. Submit data, typically to create a resource or trigger processing. The catch-all for non-idempotent actions.
  • PUT. Replace a resource entirely at a known URL.
  • PATCH. Apply a partial update to a resource.
  • DELETE. Remove a resource.
  • HEAD. Like GET but returns headers only, no body. Useful for checking existence or metadata cheaply.
  • OPTIONS. Ask which methods and capabilities a resource supports. Used heavily in CORS preflight requests.

Safe and idempotent

Two properties come up constantly, and they are precise terms, not vibes:

  • Safe means the request has no intended side effects: it only reads. GET, HEAD, and OPTIONS are safe.
  • Idempotent means sending the request once or many times leaves the server in the same resulting state. GET, HEAD, OPTIONS, PUT, and DELETE are idempotent.
MethodSafeIdempotentTypical use
GETYesYesRead a resource
HEADYesYesRead headers only
OPTIONSYesYesDiscover capabilities
PUTNoYesReplace a resource
DELETENoYesRemove a resource
POSTNoNoCreate / trigger action
PATCHNoNo*Partial update

The gotchas, which are exactly what interviews probe:

  • POST is neither safe nor idempotent. Resending it twice creates two resources, which is why a flaky network retry on a POST can double-charge a customer.
  • PUT is idempotent but POST is not. PUT sets the resource to a fully specified state, so resending lands the same state. POST appends or creates, so resending compounds.
  • PATCH is not guaranteed idempotent (the asterisk above). The spec does not require it, even though a given PATCH can be written to be idempotent (for example setting a field to an absolute value rather than incrementing it).
  • DELETE is idempotent even though the second call may return 404. Idempotency is about the resulting state of the server, not about the response code being identical. After one DELETE the resource is gone; after a second DELETE it is still gone.

Status codes

The status code is a three-digit number whose first digit gives the class of outcome:

ClassMeaning
1xxInformational (rarely seen directly)
2xxSuccess
3xxRedirection
4xxClient error (the request was wrong)
5xxServer error (the server failed)

The specific codes worth knowing cold:

  • 200 OK generic success. 201 Created a resource was created (often with a Location header pointing to it). 204 No Content success with nothing to return, common for DELETE and some PUTs.
  • 301 Moved Permanently vs 302 Found (temporary): permanent redirects can be cached and change SEO behaviour. 304 Not Modified tells the client its cached copy is still valid (used with ETags, below).
  • 400 Bad Request malformed request. 401 Unauthorized actually means unauthenticated (no or bad credentials), while 403 Forbidden means authenticated but not allowed. 404 Not Found. 409 Conflict the request clashes with current state (for example a version conflict). 422 Unprocessable Entity well-formed but semantically invalid, common for validation errors. 429 Too Many Requests you have been rate limited.
  • 500 Internal Server Error the server threw. 502 Bad Gateway and 503 Service Unavailable usually come from a proxy or load balancer when the upstream is broken or overloaded.

Headers

Headers are the key/value metadata that make HTTP flexible. The ones that earn their keep:

  • Content-Type. The media type of the body, such as application/json or text/html. The client and server both rely on it to parse the body.
  • Content-Length. The size of the body in bytes, so the receiver knows where the message ends.
  • Authorization. Carries credentials, commonly Bearer <token> for a JWT or API key.
  • Cache-Control. Directives such as no-store, max-age=3600, or private that tell clients and proxies whether and how long they may cache the response.
  • ETag and If-None-Match. An ETag is a fingerprint of a resource version. The client sends it back in If-None-Match on the next request; if it still matches, the server replies 304 Not Modified with no body, saving bandwidth. This is conditional caching.
  • Cookie and Set-Cookie. How state is bolted onto a stateless protocol, covered next.

Statelessness, cookies, and sessions

Because HTTP forgets everything between requests, anything resembling a logged-in session has to be reconstructed on every request. The classic mechanism is cookies: the server sends Set-Cookie: session=abc123 on login, the browser stores it, and the browser automatically attaches Cookie: session=abc123 to every subsequent request to that domain. The server looks up that identifier to recover who you are.

There are two broad styles. A server-side session stores the real state in the server or a shared store (like Redis) and the cookie holds only an opaque ID. A token-based approach (such as a JWT) puts the signed state in the token itself, so the server can verify it without a lookup, which keeps things stateless and scales more cleanly. Either way, the protocol stays stateless; the state lives in what the client sends back.

HTTP versions: 1.1 to 3

HTTP's semantics (methods, status codes, headers) have stayed largely stable; what has changed is how bytes move over the wire, and every step has been about killing latency and head-of-line blocking.

HTTP/1.0 and 1.1

HTTP/1.0 opened a fresh TCP connection per request, paying the handshake cost every time. HTTP/1.1 added persistent connections (keep-alive) so a connection could be reused across requests. It also specified pipelining, letting a client send several requests without waiting for each response, but pipelining failed in practice: responses had to come back in order, so one slow response stalled everything behind it. This is head-of-line (HOL) blocking at the application layer. Browsers worked around it by opening roughly six parallel TCP connections per origin.

HTTP/2

HTTP/2 replaced the text protocol with a binary framing layer and introduced multiplexing: many independent streams share a single TCP connection, with requests and responses interleaved as frames. It added HPACK header compression (headers are repetitive and were pure overhead) and a now largely deprecated server push. Multiplexing fixes application-layer HOL blocking, but there is a catch: HTTP/2 is still subject to TCP-level HOL blocking. Because all streams share one TCP connection, a single lost packet makes TCP hold back every stream until it is retransmitted, since TCP guarantees an in-order byte stream and cannot tell that the streams are independent.

HTTP/3

HTTP/3 fixes that by changing the transport underneath. It runs over QUIC, which is built on UDP rather than TCP (QUIC was mentioned in the TCP & UDP note as rebuilding reliability in user space). QUIC understands streams natively, so a lost packet only stalls its own stream and the others keep flowing, finally eliminating transport-level HOL blocking. QUIC also folds the transport and TLS handshakes into one round trip (with 0-RTT resumption for repeat connections), so connections start faster. The reliability, ordering, and congestion control that TCP used to provide are reimplemented inside QUIC on top of UDP's bare datagrams.

The through-line: HTTP/2 solved blocking at the application layer but was still held back by the very TCP guarantees that the TCP & UDP note describes, and HTTP/3 had to drop down to UDP to solve it properly.
