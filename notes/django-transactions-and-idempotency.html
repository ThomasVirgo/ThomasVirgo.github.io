Django Transactions & Idempotency

These two topics belong together. Transactions are how the database lets you group writes so they all happen or none of them do. Idempotency is the property that lets a client safely retry a request without causing damage. In a real backend they meet at exactly the point that hurts most: a payment that must be recorded once, even when the network, the browser, or a task queue tries to make it happen twice. This note covers Django's transaction machinery first, then uses it to build an idempotent POST /orders endpoint.

Part 1: Transactions in Django

Autocommit is the default

Out of the box Django runs in autocommit mode. Every ORM write is wrapped in its own transaction and committed immediately, so two saves are two independent transactions:

order.status = "paid"
order.save()            # commit 1
customer.balance -= 100
customer.save()         # commit 2

If the process dies between the two saves, the order is marked paid but the balance is never debited. That is the problem transactions exist to solve.

transaction.atomic()

transaction.atomic() opens a transaction and commits it when the block exits cleanly, or rolls it back if an exception propagates out. It works as a context manager or a decorator:

from django.db import transaction

with transaction.atomic():
    order.status = "paid"
    order.save()
    customer.balance -= 100
    customer.save()
    # both writes commit together, or neither does


@transaction.atomic
def settle(order, customer):
    ...

Nesting creates savepoints, not real transactions

Databases do not have true nested transactions. When you nest atomic() blocks, the outer block is the real transaction and each inner block is a savepoint: a marker you can roll back to without discarding the whole transaction. An inner block failing rolls back to its savepoint; the outer block can catch that and still commit everything else.

with transaction.atomic():          # outer: real transaction
    create_order()
    with transaction.atomic():      # inner: savepoint
        risky_write()               # rolls back to the savepoint on error

This is the mechanism behind the most important idempotency gotcha below, so keep it in mind.

ATOMIC_REQUESTS

Set ATOMIC_REQUESTS on a database and Django wraps every view in a transaction: the request commits if the view returns normally and rolls back if it raises.

DATABASES = {
    "default": {
        # ...
        "ATOMIC_REQUESTS": True,
    }
}

It is a blunt instrument. The transaction is held open for the entire request, including slow template rendering or external API calls, which lengthens how long locks are held. It also fights with anything that needs to commit mid-request. Prefer explicit atomic() blocks around the writes that actually need them.

Locking rows with select_for_update()

When two requests read the same row, change it, and write it back, one update can silently clobber the other. select_for_update() takes a row-level lock so the second reader blocks until the first transaction commits. It must run inside an atomic() block, because the lock is released at commit time.

with transaction.atomic():
    account = Account.objects.select_for_update().get(pk=pk)
    account.balance -= 100      # no other transaction can touch this row yet
    account.save()

Running code after commit with on_commit()

A classic bug: a view saves an order, fires a Celery task to email a receipt, and the task runs before (or instead of) the commit. If the transaction later rolls back, you have emailed a receipt for an order that does not exist. Or the worker picks up the task before the row is visible and gets a DoesNotExist. transaction.on_commit() defers a callback until the transaction actually commits, and skips it entirely on rollback.

with transaction.atomic():
    order.save()
    transaction.on_commit(lambda: send_receipt.delay(order.id))
# send_receipt is queued only if this transaction commits

A word on isolation levels

Isolation level controls what one transaction sees of another's in-flight work. On Postgres the default is READ COMMITTED: each statement sees rows committed before it started, so you can get different results from two reads in the same transaction if someone commits in between. Stricter levels (REPEATABLE READ, SERIALIZABLE) trade throughput for fewer anomalies and can raise serialization errors you must retry. For most Django apps the default plus targeted select_for_update() is enough; reach for higher isolation only when you have a specific anomaly to prevent.

Part 2: Idempotency

What idempotent means

An operation is idempotent if applying it once and applying it many times have the same effect. Setting a value to 5 is idempotent; adding 5 is not. In HTTP the spec already labels methods this way: GET, PUT, and DELETE are idempotent, and POST is not, because two POSTs are meant to create two resources. That is precisely why POST endpoints are the ones that need help.

Why it matters

You cannot avoid retries. A client whose connection drops after the server committed, but before the response arrived, has no way to know the write succeeded, so it retries. A user double-clicks the pay button. An at-least-once task queue redelivers a message. Every one of these can turn a single intended charge into two real ones. Idempotency is what makes "just retry it" a safe default instead of a way to double-bill customers.

The idempotency-key pattern

The client generates a unique key for the operation (a UUID) and sends it, usually as an Idempotency-Key header. The server records that key against the work it did. If the same key arrives again, the server recognises the replay and returns the original result instead of repeating the work. The retry has to carry the same key, which is why the client generates it, not the server.

Approach 1: a unique constraint (recommended)

The simplest robust version puts a unique constraint on the key and lets the database enforce dedup. Store the key on the resource itself:

class Order(models.Model):
    idempotency_key = models.CharField(max_length=255, unique=True)
    amount = models.DecimalField(max_digits=10, decimal_places=2)
    # ...

The view wraps the write in atomic() and relies on the unique constraint to reject the second attempt with an IntegrityError. On a replay, the row already exists, so you fetch and return it:

from django.db import IntegrityError, transaction
from django.http import JsonResponse

def create_order(request):
    key = request.headers["Idempotency-Key"]
    try:
        with transaction.atomic():
            order = Order.objects.create(idempotency_key=key, amount=amount)
            transaction.on_commit(lambda: charge_customer.delay(order.id))
    except IntegrityError:
        # key already used: the order exists, return it instead of recreating
        order = Order.objects.get(idempotency_key=key)
    return JsonResponse({"id": order.id})

The database, not your application code, is the source of truth for "has this happened before". There is no check-then-insert race window, because the uniqueness check and the insert are the same atomic operation.

Approach 2: a dedicated key table

The unique constraint dedups the write but does not remember the exact response you sent. When a client needs the identical response body on replay (the same generated fields, the same status code), store it against the key in its own table:

class IdempotencyKey(models.Model):
    key = models.CharField(max_length=255, unique=True)
    response_body = models.JSONField()
    status_code = models.PositiveSmallIntegerField()
    created_at = models.DateTimeField(auto_now_add=True)

At the top of the view you look up the key; a hit returns the stored response immediately, a miss does the work and records the response before returning. It is more moving parts, so reach for it only when replaying the exact response actually matters. For most endpoints, Approach 1 is enough.

Gotchas

Catching IntegrityError inside atomic() is a trap. Once any database error fires inside an atomic() block, the transaction is already marked for rollback. Catching the exception does not un-break it: the next query in that block raises TransactionManagementError. Catch it outside the block (as in Approach 1), or wrap just the risky write in a nested atomic() so only the savepoint rolls back and the outer transaction survives:

with transaction.atomic():
    write_audit_log()               # want to keep this
    try:
        with transaction.atomic():  # savepoint around the risky insert
            Order.objects.create(idempotency_key=key, amount=amount)
    except IntegrityError:
        order = Order.objects.get(idempotency_key=key)
    # outer transaction is still valid and commits the audit log

on_commit callbacks are silent when the transaction rolls back, and in tests. That is the point on rollback, but it surprises people writing tests: Django's TestCase wraps each test in a transaction that never commits, so on_commit callbacks never fire. Use TestCase.captureOnCommitCallbacks(execute=True) to run them, or a TransactionTestCase.

select_for_update() only works inside a transaction. In autocommit mode there is no open transaction to hold the lock, so Django raises TransactionManagementError. Always put it inside an atomic() block.

Put together, the rule of thumb is: wrap related writes in atomic(), let a unique constraint enforce idempotency, catch the resulting IntegrityError outside the block, and defer side effects to on_commit() so they only happen when the work actually stuck.
