TCP & UDP

TCP and UDP are the two dominant transport-layer (layer 4) protocols. They both run on top of IP and add the concept of ports, so a single host can host many independent conversations at once. Where IP gets a packet to the right machine, the transport layer gets it to the right application and decides how much reliability to add on top of IP's best-effort delivery.

Ports and sockets

A port is a 16-bit number (0 to 65535) identifying an application endpoint on a host. A connection is identified by the four-tuple of source IP, source port, destination IP, and destination port. Common defaults include 80 (HTTP), 443 (HTTPS), 53 (DNS), and 22 (SSH).

TCP: reliable, ordered, connection-oriented

TCP (Transmission Control Protocol) gives the application a reliable, ordered byte stream. It hides packet loss, reordering, and duplication so the receiver reads exactly what the sender wrote, in order.

The three-way handshake

A connection is established before any data flows:

Client            Server
  |  --- SYN --->    |
  |  <-- SYN/ACK --  |
  |  --- ACK --->    |
  |   (connected)    |

Teardown uses a similar exchange of FIN and ACK segments to close each direction of the stream independently.

How TCP delivers its guarantees

  • Sequence numbers. Every byte is numbered, so the receiver can reorder segments and detect gaps.
  • Acknowledgements and retransmission. The sender resends data that is not acknowledged within a timeout.
  • Flow control. A receiver-advertised window stops a fast sender from overwhelming a slow receiver.
  • Congestion control. Algorithms such as slow start and congestion avoidance back off when the network is loaded, keeping the internet as a whole stable.

UDP: minimal, message-based, connectionless

UDP (User Datagram Protocol) adds almost nothing to IP: ports and an optional checksum. Each datagram is sent independently with no handshake, no acknowledgements, and no ordering. Packets may be lost, duplicated, or arrive out of order, and the application must handle that itself.

The trade-off is speed and control. There is no connection setup latency and no head-of-line blocking, which suits applications that prefer fresh data over complete data, or that build their own reliability on top.

When to use which

  • TCP: web (HTTP/1.1 and HTTP/2), APIs, file transfer, email, SSH. Anything where correctness matters more than latency.
  • UDP: live video and voice, online games, and DNS, where a lost packet is better dropped than waited for. It is also the base for QUIC, which rebuilds reliability and ordering in user space and now underpins HTTP/3.

A useful rule of thumb: reach for TCP by default, and choose UDP only when you have a specific reason to manage delivery yourself or cannot tolerate TCP's latency.
